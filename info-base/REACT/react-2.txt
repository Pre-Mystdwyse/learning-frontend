в React тот, кто вызывает - всегдя является родителем. главным родителем всего
проекта всегда является App.jsx. в данном контексте, App.jsx вызывает Header.jsx,
передавая в него параметры hero;

при передачи данных имеет значение лишь то, как были названы элементы внутри тега
компонента, к которому ссылаюсь, т.е. если написать gold={hero.gold}, то в теге,
которому отправляются эти данные, нужно будет написать лишь ({ gold }), а само
число, прописанное в App.jsx уже закреплено за этим пропсом и будет перелано по ссылке
gold. т.е. React всегда упаковывает все передаваемые объекты в 1 объект:
    если передавать <Header hero='Карбел', status='активен' />, то React создаст объект
    const props = { hero: 'Карбел', status: 'активен' };, отчего дальшнейшее написание
    будет не очен удобным:
        function Header(props) {
            return <h1>{props.hero}</h1> //придётся везде писать props.
        }

альтернативой может быть вариант передачи всего объекта, чтобы уже в дочернем элементе
взять всё, что нужно. в этом случае будет нужно в App.jsx написать <Header hero={hero} />,
принимать по названию параметра тега: function Header({ hero }) и забирать необходимые
элементы обхекта: <header> Золото: {hero.gold}, Цена: {hero.price} </header>;

почему в React везде используются фигурные скобки?:
    внутри JSX (разметки React) фигурные скобки - это сигнал для React, что внутри пишется
    JS-код, а не просто обычный текст:
        - name='Карбел' - скобки не нужны, ибо это простая строка (текст);
        - age={25} - скобки нужны, ибо 25 - это число (тип данных JS);
        - isAdmin={true} - скобки нужны, ибо это булево значение;
        - user={{ name: 'Карбел' }} - тут двойные скобки. внешние означают, что внутри
        JS-код, а внутренние создают обычный JS-объект;

props.children:
    обычные пропсы передаются как атрибуты внутри открывающего тега: <Component name='Карбел' />;
    у этого тега нет закрывающего брата;
    но иногда нужно создать компонент-оболочку (например, рамку, модальное окно или карточку),
    внутри которого нужно поместить другие теги:
        //родительский компонент
        <Card>
            <h1>Привет</h1>
            <p>Это текст внутри карточки</p>
        </Card>
    Card открыта и закрыта. всё, что находится внутри между ними, автоматически попадает
    в специальный проп под названием children;
    внутри самого компонента пишется:
        function Card({ children }) {
            return (
                <div className="border-style">
                    {children} {/*сюда автоматически вставляется h1 и p из родительского */}
                </div>
            );
        }
    props.children нужен для создания гибких компонентов-контейнеров;

данные в React текут сверху вниз (от родителя к жочерним элементам), но когда нужно поменять
какие-то данные в интерфейсе 9например, по клику по кнопке, тут помогут состояние и коллбек:
    если ребёнку нужно изменить что-то у родителя, то родитлеь должен передать ребёнку
    "пульт управления" в виде функции:
        1. в App.jsx создаётся переменная состояние name и функция её изменения setName;
        2. App передаёт эту функцию внутрь дочернего компонента через пропсы;
        3. дочерний компонено вызывает эту функцию при клике;
        пример кода:
            //родитель App.jsx
            function App() {
                const [userName, setUserName] = useState('Карбел')

                return (
                    <div>
                        <h1>Текущий пользователь: {userName}</h1>
                        {/*передаём функцию setUserName как проп под именем changeName */}
                        <Button changeName={setUserName} />
                    </div>
                );
            }

            //дочерний компонент Button.jsx
            function Button({ changeName }) {
                return (
                    //при клике вызывается функция родителя, и имя в App изменится на 'Аметист'
                    <button onClick={() => changeName('Аметист')}>
                        Сменить пользователя
                    </button>
                );
            }

вложенные функции:
    герой нужно купить предметы. в родителе хранится золото героя, а дочерний компонент -
    это кнопка магазина;
    вместо того, чтобы отдавать дочернему компоненту прямой пульт от золота, напишем функцию
    buyItem. в ней сначала проверим, хватает ли у игрока денег:
        import { useState } from 'react';
        import ShopItem from './ShopItem';

        function App() {
            const [gold, setGold ] = useState(100);

            function handleBuy(price) {
                if (gold >=price) {
                    setGold(gold - price);
                    alert('Предмет успешно куплен!');
                } else {
                    alert('Нужно больше золота!');
                }
            }

            return (
                <div style={{ padding: '20px' }}>
                    <h1>Текущее золото героя: {gold}</h1>

                    {/* передаём функцию handleBuy в проп под названием onBuy */}
                    <ShopItem title='Меч правосудия' price={40} onBuy={handleBuy} />
                    <ShopItem title='Зелье здоровья' price={120} onBuy={handleBuy} />
                </div>
            );
        }

    дочерний компонент ничего не знает про общее количество золота в App.jsx. он просто
    знает свою цену и имеет кнопку связи onBuy, которую ему передал родитель:
        function ShopItem({ title, price, onBuy }) {
            return (
                <div style={{ border: '1px solid gray', margin: '10px', padding: '10px' }}>
                    <h3>{title}</h3>
                    <p>Цена: {price}</p>

                    {/* при клике вызываем функцию родитля и передаём в неё цену этого предмета */}
                    <button onClick={() => onBuy(price)}>
                        Купить
                    </button>
                </div>
            );
        }

    фишки такого подхода:
        1. безопасность данных (инкапсуляция): магазин (ShopItem) - это просто кнопка. он
        не может сам забрать золота или накрутить его себе. вся важная логика и само состояние
        денег защищены внутри родителя (App);
        2. переиспользование: использован один и тот же компонент ShopItem дважды - для меча и
        для зелья. они оба работают через одну и ту же функцию родителя, но передают в неё
        разные цены;
    в примере выше функция handleBuy была написана внутри функции-компонента App. в JS это
    называется вложенными функциями (или замыканием если внутренняя функцияч использует
    переменные внешней);
    вот в чём заключается главный практический смысл такого подхода:
        1. доступ к переменным родительской функции. внутренняя функция имеет автоматический
        доступ ко всему, что находится во внешней функции (к переменным, пропсам, стейту);
        если бы мы вынесли handleBuy за пределы App, то нам бы пришлось каждый раз передавать
        туда текущее золото: handleBuy(price, gold, setGold). это неудобно и усложняет код;
        2. инкапсуляция. если функция нужна лишь только внутри одного конкретного компонента,
        нет смысла выносить её наружу или делать глобальной:
            - писать функцию внутри компонента - это способ сказать, что эта логика уникальна
            для этого места, больше никот в приложении её использовать не должен;
            - это предотвращает засорение общей памяти приложения;
        3. создание специфических обработчиков (хэндлеров). в React часто вложенные функции
        используются для обработки событий (клик, ввод текста, отправка формы). их принято
        называть с приставкой handle (например, handleSubmit, handleInputChange);
            function SearchForm() {
                const [query, setQuery ] = useState("");

                //внутренняя функция для обработки отправки формы
                function hanldeSubmit(event) {
                    event.preventDefault(); //отменяем перезаргузку страницы браузером
                    console.log('ищем в базе данных: ', query);
                }

                return (
                    <form onSubmit={handleSubmit}>
                        <input value={query} inChange={(e) => setQuery(e.target.value)} />
                        <button type="submit">Найти</button>
                    </form>
                );
            }

        здесь функция handleSubmit создана специально для этой формы. ей не нужно быть снаружи,
        так как её единственная задача - сработать при нажатии кнопки "Найти" и прочитать стейт
        query;

        в <input value={query} inChange={(e) => setQuery(e.target.value)} /> e (или любое другое
        название переменной, но принято e в стрелочной функции) - это специальный объект, который
        автоматически создаёт сам браузер (и React) в тот момент, когда пользователь совершает
        какое-то действие на странице. в данном случае действие - это нажатие клавиш в поле
        ввода (input);
        внутри этого объекта e хранится гигантское количество полезной информации о произошедшем
        событии:
            - какая клавиша была нажата;
            - в какую микросекунду это произошло;
            - координаты курсора мыши;
            - на каком именно элементе это событие случилось;
        что значит .target.value?
        это просто путь внутри объекта e, по которому мы добираемся до текста, написанного в
        инпуте:
            1. e - сам объект события;
            2. e.target - свойство внутри объекта, которое указывает на цель события. то есть на
            конкретный HTML-тег, с которым взаимодействовал пользователь. в нашем случае
            e/target - это сам тег <input>;
            3. e.target.value - это текущее текстовое значение (то, что физически напечатано
            в данный момент внутри этого инпута);

        метод event.preventDefault() (предотвратить дефольное действие) нужен для того,
        чтобы отменять стандартное поведение браузера, которое заложено в него по умолчанию
        для определённых событий;
        в случае с формами (<form>) у браузера есть очень старое поведение, которое тянется
        ещё из 90-ых годов, со времён зарождения интернета;

        когда пользователь нажимает кнопку <button type="submit"> или жмёт Enter внутри инпута,
        браузер считает своей главной обязанностью:
            1. собрать все данные из полей формы;
            2. отправить их на сервер;
            3. полностью перезагрузить страницу;
        
        почему в React это огромная проблема?
        React - это библиотека для создания SPA (Single Page Application - одностраничных приложений);
        в React вся магия происходит прямо в браузере за счёт JS. весь стейт приложения,
        введённые пользователем данные, открытые вкладки - всё это живёт в оперативной памяти
        страницы;
        если страница полностью перезагрузится:
            - весь JS-код запустится с чистого листа;
            - все стостояния сбросятся к начальным значениям;
            - экран на долю секунды моргнёт белым цветом;
        вместо грубой перезагрузки мы хотим, чтобы React сам, тихо и незаметно, обработал данные
        (например, проверил их валидность, сохранил в стейт или отправил через фоновый запрос
        fetch/axios);

        как event.preventDefault() спасает ситуацию:
            когда мы пишем:
                function handleSubmit(event) {
                    event.preventDefault();
                    console.log('Ишем в базе данных: ', query);
                }
            мы буквально запрещаем браузеру обновлять страницу в этот момент;
            браузер останавливает отправку формы, страница не перезагружается, а код,
            написанный ниже спокойно выполняется;

        где ещё используется preventDefault()?
        этот метод применяется везде, где стандартное поведение браузера мешает
        логике приложения:
            1. на ссылках (<a href="...">);
            по умолчанию клик по ссылке переносит пользователя на другой URL с перезагрузкой
            страницы. если нужно сделать кнопку-ссылку, которая просто открывает модальное
            окно на этой же странице, то нужно написать:
                function handleClick(e) {
                    e.preventDefault();
                    setModalOpen(true);
                }
            2. при отправке файлов (Drag and Drop);
            если перетащить картинку с рабочего стола в окно браузера, он по умолчанию
            пытается просто открыть этот файл в текущей вкладке. чтобы сделать красивую
            зону для загрузки файлов, разработчики отменяют это поведение через
            e.preventDefault();
